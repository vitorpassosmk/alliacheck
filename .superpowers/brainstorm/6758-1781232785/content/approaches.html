Como reestruturar o fluxo do frete?

3 abordagens para implementar o novo pipeline de estados

A

Refatoração limpa Recomendado

Substituir os estados CT-e (AGUARDANDO_NF, NF_RECEBIDA, CT_E_EMITIDO) por novos estados na viagem principal: AGUARDANDO_CTE e CTE_EMITIDO. Remover completamente o campo cte_status do banco. O número do CT-e (44 dígitos) fica salvo diretamente na tabela fretes. Remover upload de NF-e, checklist de 6 itens e toda a aba de documentos.

Prós

  • Código limpo, sem estados mortos
  • Fluxo único e sem ambiguidade
  • Kanban reflete exatamente a realidade
  • Menos superfície de bug

Contras

  • Migration no banco necessária
  • Maior escopo de mudança
B

Abordagem híbrida (manter cte_status)

Manter cte_status como campo auxiliar, mas esconder da UI. Adicionar os novos estados de viagem por cima. O estado AGUARDANDO_CTE na viagem seria sincronizado com cte_status internamente.

Prós

  • Menos risco de quebrar algo existente
  • Reversível mais facilmente

Contras

  • Dois sistemas de estado paralelos — confuso
  • Lógica duplicada no backend
  • Deixa código morto no projeto
C

Feature flag / modo simplificado

Criar um toggle por cliente que ativa o "modo simplificado" (sem NF-e, sem checklist). Os dois fluxos coexistem no código, ativados por configuração do tenant.

Prós

  • Flexível para outros clientes futuros

Contras

  • Complexidade prematura — só há um cliente agora
  • Dobra o código de fluxo a manter
  • YAGNI — não há requisito multi-fluxo hoje

Recomendação: Opção A — refatoração limpa. O projeto ainda está em fase pré-demo, o banco ainda não está em produção com dados reais, e código limpo vai facilitar tudo daqui pra frente. A Opção B deixa dívida técnica desde o início; a C é prematura.
